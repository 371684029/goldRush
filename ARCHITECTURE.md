# GoldRush 黄金投资研究 Agent — 架构说明文档

## 0. 写在前面

这份文档记录了 GoldRush 从想法到完整架构的来龙去脉。不是枯燥的规格书，而是一份"为什么这样设计"的决策记录。

---

## 1. 起点：要解决什么问题？

我通过支付宝基金平台做黄金投资，偏中长期配置（定投+波段），但也想关注短期波动。每次做决策要翻很多信息：

- 金价怎么样了？美元指数呢？美债呢？
- 黄金基金（000216/002610）现在估值贵不贵？该定投加码还是暂停？
- 技术面什么信号？基本面什么逻辑？情绪面呢？
- 这些信息散落在不同平台，而且中文源和英文源各有盲区

**核心痛点**：信息碎片化 + 缺乏系统化分析 + 中英信息茧房。

**期望**：一个命令，自动搜集数据、交叉验证、给出短期+中长期双视角的分析报告。

## 2. 关键决策记录

### 决策 1：独立 CLI 应用 vs opencode skill

| 选项 | 优势 | 劣势 |
|------|------|------|
| opencode skill | 无需额外 LLM 调用 | 独立性差，只能在 opencode 里用 |
| 独立 CLI 应用 | 完全独立，可分发 | 需要 LLM 调用方案 |

**结论：独立 CLI 应用。** 可以独立运行、分发、持续迭代，不绑定 opencode 环境。

### 决策 2：LLM 调用方式

| 选项 | 说明 |
|------|------|
| opencode CLI 子进程 | `opencode run -m glm-5.1 "..."` — 测试发现报错 "Session not found"，不可靠 |
| opencode SDK (`@opencode-ai/sdk`) | TypeScript 原生 SDK，支持 session.prompt + 结构化输出 + websearch — **选定** |
| 直接 HTTP API | 需要确认 API endpoint，SDK 已经封装好了 |

**结论：用 `@opencode-ai/sdk`。** 原生 TypeScript、支持结构化输出、自带 websearch。

### 决策 3：LLM 计费方案

| 选项 | 说明 |
|------|------|
| opencode Zen（按量付费） | 每个 token 扣钱，高频使用成本不可控 |
| opencode Go（月费 $10） | 封顶，$60 月额度，每天跑几次绰绰有余 — **选定** |

**结论：opencode Go 套餐。** 每次 analysis 要 5+ 次 LLM 调用，按量付费会很贵。

### 决策 4：搜索引擎 — 为什么需要 Exa？

初始方案只用 opencode 内置 websearch，但发现：

| 问题 | opencode websearch | Exa API |
|------|-------------------|---------|
| 英文金融数据（美联储/COMEX） | 中文搜索引擎为主，英文覆盖弱 | 有 `financial report` 分类，专业 |
| 结构化输出 | 需 LLM 从网页提取 | 原生支持 JSON Schema |
| Token 效率 | 返回全文，LLM 自己提炼 | Highlights 10x 压缩，只返回相关片段 |
| 中文数据（上海金/基金净值） | ✅ 覆盖好 | ❌ 覆盖弱 |

**结论：双引擎互补。** Exa 负责英文/金融数据，opencode websearch 负责中文数据，关键数据双搜交叉验证。

### 决策 5：为什么要本地历史数据？

纯靠搜索有个致命问题：**技术指标算不了。**

| 指标 | 需要的历史数据 | 纯搜索能拿到吗 |
|------|---------------|--------------|
| MA20 | 最近20天收盘价 | ❌ 搜不到完整序列 |
| RSI(14) | 最近15天以上涨跌 | ❌ |
| MACD | 需要26天以上 | ❌ |
| 布林带 | 20天收盘+标准差 | ❌ |
| 估值百分位 | 1年以上历史 | ❌ |

**结论：用 SQLite 存本地快照。** 每次运行自动存数据，首次用 `init-history` 拉取60天历史，指标本地计算后注入 prompt。

### 决策 6：投资视角 — 短期 + 中长期双轨

一开始只做中长期（周线、定投），但实际投资中短期信号也很重要：

- 金价日线超买但周线看多 → 短期可能回调，中长期继续持有
- 这两个信号矛盾时 → 都展示出来，让用户自己判断

**结论：技术面同时输出短期（日线）和中长期（周线）两套指标，综合研判给出两套策略。**

用 `--horizon short|mid|all` 控制输出范围。

### 决策 7：信息可靠性 — 怎么防"幻觉"？

LLM 天然会编造数据。我们的防线：

1. **来源分级**：A级（交易所/央行）> B级（财经媒体）> C级（自媒体）
2. **3源交叉验证**：同一数据点至少搜3个独立来源，2+源一致才采信
3. **中英文双搜**：避开单一信息茧房
4. **反向核查**：重大新闻必须搜反对观点
5. **时效标注**：每个数据标注获取时间和来源

## 5. LLM 调用链路

```
用户执行 goldrush analysis
        │
        ▼
   opencode SDK 创建 session
        │
        ├──→ 数据采集Agent (deepseek-v4-flash)
        │     · 6-8 组搜索（Exa + opencode 按路由分发）
        │     · 结构化提取 MarketData
        │     · ~2-3 次 LLM 调用
        │
        ├──→ 验证Agent (deepseek-v4-flash)
        │     · 2-3 组验证搜索
        │     · 交叉比对 + 来源分级
        │     · ~1-2 次 LLM 调用
        │
        ├──→ 技术面Agent (glm-5.1)
        │     · 注入本地计算的技术指标
        │     · 注入搜索到的市场解读
        │     · 1 次 LLM 调用（structured output）
        │
        ├──→ 基本面Agent (glm-5.1)
        │     · 注入宏观数据
        │     · 含反向搜索的论据
        │     · 1 次 LLM 调用
        │
        ├──→ 情绪面Agent (glm-5.1)
        │     · 注入情绪面数据
        │     · 1 次 LLM 调用
        │
        ├──→ 基金面Agent (glm-5.1)
        │     · 注入基金净值/费率/溢价数据
        │     · 1 次 LLM 调用
        │
        └──→ 综合编排Agent (glm-5.1)
              · 汇总四维度 + 市场数据 + 历史趋势
              · 1 次 LLM 调用
              · 输出双轨策略报告

   总计 LLM 调用：~8-10 次
   总计搜索请求：~10-15 次（Exa + opencode）
```

## 7. 双视角策略设计

### 为什么需要双视角？

```
现实中的矛盾场景：

金价日线超买（RSI>70）     →  短期：可能回调，建议减仓
金价周线站在20周均线上方    →  中长期：趋势健康，继续持有

如果只看短期 → 可能错过长期上涨
如果只看中长期 → 可能忽视短期风险

解决方案：同时输出两套分析，让用户根据自身情况选择
```

### 短期视角（日线级别）

```
┌─────────────────────────────────┐

│                                 │
```

## 8. 核心品种对照

```
短期品种（场内，T+0）         中长期品种（场外，支付宝）
─────────────────────         ────────────────────────
518880 华安黄金ETF            000216 华安黄金易A（长持>1年）
518860 博时黄金ETF            000217 华安黄金易C（波段<1年）
纸黄金（银行）                002610 博时黄金A（长持>1年）
                              002611 博时黄金C（波段<1年）
                              积存金（支付宝，小额定投入门）

联系：场内ETF净值 ≈ 场外联接基金净值（跟踪同一标的）
区别：
  · 场内ETF 可实时交易，适合短线
  · 场外基金 15:00前申购按当日净值，适合定投
  · A类有申购费无销售服务费，适合长期
  · C类无申购费有销售服务费，适合短期波段
```

## 12. 风险与待确认项

| # | 风险/待确认 | 影响 | 应对方案 |
|---|------------|------|---------|
| 1 | opencode SDK 日志级别不确定 | CLI 输出可能被 server 日志污染 | 测试后决定是否需要静默模式 |
| 2 | SDK session 是否支持并行 prompt | 影响四维度分析是否可并行 | 如不支持则串行执行，增加延迟 |
| 3 | Exa 中文金融数据覆盖质量 | 上海金/基金净值可能搜不好 | 中文数据 fallback 到 opencode |
| 4 | Exa 免费额度（1000/月） | 每次分析约8-12次搜索，月约80次可覆盖 | 控制搜索频率，配搜索缓存 |
| 5 | opencode Go 套餐 5h/$12 周限额 | 高频使用可能触发限速 | 配置调用预算管理 |
| 6 | 支付宝基金数据获取 | 天天基金网页可能有反爬 | 优先 websearch，备选 API |
| 7 | 历史数据初始化 | 60天历史需要较多搜索额度 | 分批拉取，错峰执行 |
